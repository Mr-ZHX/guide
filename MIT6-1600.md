# MIT 6.1600 Labs — Overview (English)

This document summarizes **what each lab in the `6.1600-labs` repository is for**, using the **bundled course handouts** under `docs/lab0.md`–`lab5.md` and the **matching code directories**. It is meant for readers who do **not** know **MIT 6.1600** (Foundations of Computer Security). It states **learning goals and deliverables**, not step-by-step solutions, and does **not** include student write-ups.

**Official context:** The `docs/` pages describe **Lab 0–Lab 5** for [6.1600](https://61600.csail.mit.edu/) (starter code hosted at `github.com/mit-pdos/6.1600-labs`). Submission is typically via **Gradescope**; memory/time limits and collaboration rules are in those handouts.

## Lab 0 — Password cracking & hash basics (`hash/`)

**Handout:** `docs/lab0.md` — *“Lab 0: Password cracking”*

**Code / templates:** `hash/sol.py`, `hash/hashall.py`, `hash/hashbig.py`, `hash/grader.py`, LaTeX `hash/LaTeX/questions.tex`.

### What you do (by problem theme)

1. **Conceptual (written):** How **`/etc/shadow`** stores **hashes** (not plaintext), why that matters, **per-user hash parameters**, **PBKDF2**-style iteration and how you’d set iteration counts for web vs laptop login.

2. **Cracking (code, toy 48-bit hash):** Implement preimage search in **`sol.py`** — recover a **personal lowercase-word password** from a Gradescope-given target, crack one entry from a large **unsalted** hash list (`hashes.txt`), and answer how **salting** changes attack cost. **No** off-the-shelf password crackers.

3. **Birthday bounds (written + small code):** Balls-in-bins style **probability** exercises; collision probability for a **random function** (union bound).

4. **Collision finding (code):** For a different **56-bit** construction in `hashbig.py`, explain **cycle-based** collision finding, then implement **`problem_4b()`** to output a collision within course time/memory limits (Floyd-style / low-memory ideas per hints).

**Outcomes:** practical sense of **password hashing**, **search cost**, **salts**, and **birthday/collision** reasoning.

## Lab 1 — Merkle trees & authenticated storage (`merkle/`)

**Handout:** `docs/lab1.md` — *“Lab 1: Merkle trees”*

**Code:** `merkle/store.py`, `merkle/client.py`, `merkle/common.py`, student **`merkle/attack.py`**, CLI `merkle/cli.py`, `merkle/grader.py`.

### What you do

- **Warm-up (written):** Behavior of the **key-value store** across **server restart** and why **puts** can fail until state is consistent.

- **Understanding:** Client authenticates server replies with a **Merkle tree** over key-value data (`H_kv`, `H_internal`, `H_empty`, key **traversal paths** per handout).

- **Attack (code):** Implement attacks in **`attack.py`** that break the **integrity** goals of this authenticated store (exact attack goals are specified in the full PDF/handout; graded by `grader.py`).

**Outcomes:** see how **Merkle proofs** are supposed to prevent cheating servers, and how mistakes enable **attacks**.

## Lab 2 — Bad randomness (`bad-random/`)

**Handout:** `docs/lab2.md` — *“Lab 2: Bad randomness”*

**Code:** `bad-random/ecdsa/` (`keygen.py`, **`sol.py`**), `bad-random/wep/` (**`attacker.py`**, `victim.py`).

### What you do

- **True/false / theory:** Gradescope questions on randomness in crypto.

- **ECDSA key recovery:** `keygen.py` uses **predictable nonces**; implement functions in **`ecdsa/sol.py`** to recover **secret keys** from **public keys** and related scenarios (dates, multiple signatures) as specified.

- **WEP-style attack:** Complete **`wep/attacker.py`** to break a toy **WEP**-like wireless encryption scenario.

**Outcomes:** concrete appreciation that **bad RNG** breaks **ECDSA** and **stream ciphers** in the wild.

## Lab 3 — Performance & side channels (`timing/`, `ssh/`)

**Handout:** `docs/lab3.md` — *“Lab 3: Performance and side channels”*

**Code:** `timing/attacker.py`, `timing/bad_server.py`, … ; `ssh/attack.py`, `ssh/server.py`, `ssh/grader.py`.

### What you do

1. **Performance (written):** Questions using **OpenSSL** benchmarks — e.g. **MAC** choice for large files, **AES-128 vs AES-256** cost, relative speed of **RSA / DSA / ECDSA** for sign/verify/keygen at “128-bit security.”

2. **Timing attack (code):** Implement **`steal_password`** in `timing/attacker.py` to extract a **hex-encoded secret** from **`BadServer`** via **timing leaks**, respecting interface rules (no reading private fields), length parameter `l`, and time limits.

3. **SSH attacks (code):** Implement two attacks in **`ssh/attack.py`**:
   - **(a)** **Compression + encryption** interaction: chosen **`evil`** prefix to learn a **`secret`** JSON of city names (CRIME/BREACH-style lesson).
   - **(b)** A second attack on the provided **paramiko**-based setup (see handout; optional deterministic testing on Gradescope “3A” slot).

**Outcomes:** link **crypto performance** choices to systems, and exploit **side channels** and **protocol layering** mistakes.

## Lab 4 — Isolation (`escape/`, `wasm/`)

**Handout:** `docs/lab4.md` — *“Lab 4: Isolation”*

**Code:** `escape/grader.py`, **`escape/attack.py`**; `wasm/wasi.py`, `wasm/interactive.py`, `wasm/tinysh.c` / WASM binaries, `wasm/Makefile`.

### What you do

1. **Python “sandbox” puzzles:** Implement **`attack_one` … `attack_four`** in `escape/attack.py` to recover a **`secret`** value chosen by `grader.py` — illustrates **lack of isolation** between supposedly separate code (introspection, GC, shared state, etc.).

2. **WebAssembly sandbox escape:** Run **`tinysh`** in a **Python WASM** interpreter with a **buggy WASI** filesystem layer (`wasi.py`). Craft **shell commands** that break path confinement and read **`secret.txt`** outside the sandbox root.

**Outcomes:** understand **what isolation must guarantee** and how **small bugs** void it.

## Lab 5 — Fuzzing (`fuzz/`)

**Handout:** `docs/lab5.md` — *“Lab 5: Fuzzing”*

**Code:** **`fuzz/msgpacker.py`**, `fuzz/msgpacker_example.py`, **`fuzz/codec.py`**, `fuzz/Makefile`, Atheris-based workflow.

### What you do

1. **Find and fix bugs:** Use **fuzzing** (e.g. **Google Atheris**) to harden a deliberately buggy **MessagePack** encoder/decoder until it matches the real **MessagePack** spec (floats may be out of scope per handout). Submit fixed **`msgpacker.py`** plus evidence of fuzzing.

2. **Custom printable codec:** Implement **`encode` / `decode`** in `codec.py` mapping **bytes → bytes** with **printable-only** output, **bijection** on alphanumeric inputs, **length** constraints, and **prefix/suffix truncation** recovery properties spelled out in the handout.

**Outcomes:** use **coverage-guided fuzzing** for robustness and practice **precise specification-driven** coding.
