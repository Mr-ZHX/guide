# NanoDB-Base — Lab / Assignment Overview (English)

This document helps a reader who has **never seen the course** understand **what work this repository is organized around**, using **only artifacts present in the tree**: package layout, `testng.xml`, and the **official design-document templates** under `doc/` (these templates name each assignment and its themes; they are **not** student write-ups).

It describes **goals and functional areas**, not algorithms or implementation steps.

---

## What this codebase is

- **NanoDB** is a teaching **relational DBMS** implemented in **Java** (see `src/main/java/overview.html` and `pom.xml`: artifact `nanodb-server`, Caltech NanoDB URL).
- The project includes **parsing**, **query planning/evaluation**, **storage** (heap files, B+ tree files, buffer manager), **indexes**, **integrity constraints**, and **transactions / write-ahead logging**, with a large **TestNG** suite under `src/test/java`.

---

## How “labs” appear in *this* repository

1. **Numbered assignments (Labs 1–7)**  
   The `doc/` folder contains paired templates for each milestone:
   - `labNinfo.txt` — administrative submission form (team, hours, git tags, late tokens).
   - `labNdesign.txt` — **design questionnaire** for **CS122 Assignment *N***.  
   The **first line** of each `labNdesign.txt` states the **official assignment title** used in this courseware. Those titles are summarized below.

2. **Automated test bundles in `testng.xml`**  
   Most of the suite is **commented out** by default. The file still documents **named test groups** that map coursework to concrete classes, e.g.:
   - `assignment1-tests` → heap table format tests (`TestHeapTableFormat`).
   - `assignment2-tests` → broad SQL and built-in **function** tests (`edu.caltech.test.nanodb.sql`, `...functions`).
   - `assignment6-tests` → selected **B+ tree** and **index/unique constraint** tests (`TestBTreeFile`, `TestIndexOps`, `TestUniqueOps` methods listed in the file).

   Other tests (buffer manager, parser, expressions, full SQL features, etc.) exist under `src/test/java` and are part of the overall system even when not wired into an active `<test>` block in `testng.xml`.

## Assignments 1–7 (from `doc/lab*design.txt` titles + themes)

The following mirrors **Caltech CS122 / NanoDB** milestones as **named in the repository’s design templates**. Exact grading, files to edit, and deadlines belong to the **course handout**, not this file.

### Assignment 1 — “NanoDB Set-Up and Storage Layer”

**Themes (from the design template sections):** heap **storage** and **slotted pages**; **NULL** vs non-NULL column updates; **pinning/unpinning** tuples and pages in plans and commands; **insert performance** and block-level organization.

**What you are building toward (outcomes):** a correct **heap-file / page** representation for tuples, stable **memory management** around pages and tuples, and **mutating** tuple contents (including NULLs) on disk-backed pages.

---

### Assignment 2 — “SQL Translation and Joins”

**Themes:** turning parsed SQL into an **execution plan** (planner structure, plan simplification); **join** algorithms and planning; broader **SQL execution** behavior covered by tests in the `sql` and related areas of the codebase.

**Outcomes:** end-to-end **query planning** for non-trivial SQL (including **joins**), consistent with the course’s planner architecture (`SimplePlanner` and related code paths appear in the template questions).

---

### Assignment 3 — “Table Statistics and Plan Costing”

**Themes:** **table statistics** collection (e.g. analyzing heap files); **cost estimation** for plan nodes (scans, filters, nested-loop joins); updating statistics under comparisons; using **EXPLAIN**-style output to reason about plans on sample data.

**Outcomes:** the optimizer/planner can use **collected statistics** to assign **estimated cardinalities and costs** to plans.

---

### Assignment 4 — “Join Optimization”

**Themes:** **refactoring** planner code (shared base class between simple and cost-based planners); **cost-based join ordering** and predicate handling; **outer join** considerations (e.g. when predicates may be pushed).

**Outcomes:** a **cost-based join planner** that improves join order and access paths compared to the simpler planner from earlier assignments.

---

### Assignment 5 — “Advanced Subqueries”

**Themes:** **subquery planning** integrated into the planner; detection of subqueries in clauses such as **GROUP BY** / **ORDER BY**; **correlated** subquery evaluation.

**Outcomes:** the system can **plan and execute** advanced SQL with **subqueries**, including correlated forms, without incorrect interference between clauses.

---

### Assignment 6 — “B+ Tree Indexes”

**Themes (design template + `testng` excerpt):** **B+ tree** page layout and key sizes; **insert/delete** behavior across tree levels; **index** DDL behavior; **UNIQUE** indexes and **primary key** style constraints in operation.

**Outcomes:** persistent **B+ tree index files** that stay valid under inserts/deletes, integrated with **index creation** and **uniqueness** enforcement.

---

### Assignment 7 — “Write-Ahead Logging”

**Themes:** **WAL** correctness (`forceWAL`, ordering relative to dirty pages); **transaction manager** responsibilities; **recovery** concepts; auxiliary state such as **`txnstate.dat`** for transaction ID sequencing on restart.

**Outcomes:** **durable, crash-safe** updates: log records and data-page writes obey **write-ahead** rules; recovery can bring the database to a consistent state after failure.

## Practical notes for readers of this repo

- **`doc/lab*design.txt`** files are **questionnaires** about your design choices; they **name** each assignment’s topics but are still not a substitute for the full **PDF/web spec** from the instructor.
- **`nanodb-base`** suggests a **starting codebase**: many classes are complete, while others contain **TODOs** or **unsupported operations** until the corresponding assignment work is finished.
- To see which automated checks align with a given milestone, consult **`testng.xml`** (active and commented blocks) and browse **`src/test/java/edu/caltech/test/nanodb/...`** by subsystem.
