## Technology stack (from `package.json` and `index.html`)

- **React** (function components) with **Vite** as the build/dev server.
- **Bootstrap** and **React-Bootstrap** for layout and UI widgets (e.g. cards, buttons).
- **ESLint** configuration for basic code quality.
- The HTML entry loads a **CS571 “BadgerGuard”** script from the course API host, which ties the app to **course identity** for authenticated API use (see below).

## Domain and data

The app is branded **“Badger Recipes”** and works with **recipe objects** shaped like JSON from the CS571 ICE endpoints (comments in the starter point to paths such as `.../rest/s25/ice/pizza`, `.../pasta`, `.../chili`). Each recipe object includes fields such as:

- **Identity:** `name`, `author`
- **Media:** `img.location` (image URL), `img.description` (accessibility-oriented description)
- **Tags:** `keywords` (array of strings)
- **Social proof:** `reviews` (text + numeric ratings)
- **Ingredients:** structured amounts/units (and optional `misc` notes)
- **Instructions:** `recipe` as an ordered list of step strings

The starter embeds **example JSON** for three recipes (pizza, pasta, chili) for reference; the intended product is a **working UI** driven by that kind of data, not merely static text.

## What you are expected to accomplish (outcomes, not steps)

These are **functional goals** inferred from the scaffold and the presence of a reference `solution/` (without prescribing algorithms or code structure).

### 1. Page shell (`AllRecipes`)

- Present a clear **welcome/header** for the recipe site.
- Eventually show **multiple recipes** (the data model and API paths refer to **three** distinct recipes).

### 2. Single-recipe presentation (`Recipe`)

- Turn the placeholder “I am a recipe!” into a **real recipe card** that, when given a recipe object, shows meaningful information drawn from the object’s fields.
- A complete solution typically includes at least:
  - The **hero image** with appropriate **alt text**.
  - **Title** and **author** (and related metadata such as **keywords**).
  - A simple **interactive** element such as a **per-recipe “like” counter** that updates on user action.

### 3. Data sourcing and loading behavior

- Move from in-file examples toward **loading live recipe data** from the **CS571 REST ICE endpoints**, using the course’s expected **request identity** (via the BadgerGuard / `CS571` integration implied by the starter’s API comments and the reference app behavior).
- Handle the fact that network data arrives **asynchronously**: the UI should behave sensibly while a recipe is **not yet available** (e.g. a loading or empty state rather than broken layout).

### 4. Integration and quality

- The app should remain a standard Vite project: **`npm run dev`** for development, **`npm run build`** for production bundling, and **`npm run lint`** where required by course workflow.
- Styling should stay consistent with **Bootstrap / React-Bootstrap** usage already wired in the scaffold.
